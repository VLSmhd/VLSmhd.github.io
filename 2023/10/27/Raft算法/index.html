Raft算法

|阅读量:

raft consensus图解

动画

问题引入：假设有一个客户端、一个服务端，当客户端向服务端同步消息，在服务端单节点的情况下，很容易同步消息，服务端收到啥，这个服务端就做对应处理即可，但是当服务器用多个结点，如何保证消息、或者说信息的一致性？

解决这类问题，就用到分布式共识算法，raft就是其中之一。

算法流程

结点的状态：

  • follows追随者状态
  • candidate候选人状态
  • leader集群之首：客户端的请求优先发送到处于该状态的结点，随后同步到其它结点

选举leader

大致流程：如果一个结点的状态是follows，并且没有监听任何一个leader，那这个结点可以更换状态为candidate。随后它向周围的结点拉票，随后周围的结点就会向这个结点2投票（赞成 or 反对），当这个结点得到的票数占据大多数，就会成功选举为leader。

在raft算法中，有两个控制选举的超时设置：

  • Election timeout：followers 变成 candidate 之前等待的时间 设置在150 - 300ms之间
  • heartbeat timeout：leader结点发送（Append Entries）的间隔时间

C结点先于其它结点过完一个Election timeout，随后会开启一轮选举。C将投票请求发送给相邻结点，在这一轮中，没有进行过投票的结点会把票投给C，随后重置这些投票结点的Election timeout。

C结点获得的投票数占据大多数，选举成为leader。

心跳检测

随后C结点向相邻结点发送（Append Entries），leader发送消息（Append Entries）的时候，会按照心跳超时时间间隔发送消息（Append Entries），随后followers回馈消息，在followers接收到Append Entries时候，刷新自己的Election timeout时间。

本次选举周期会持续到追随者停止心跳，变成候选人

leader挂掉的情况：

这样即使leader结点挂掉，其它结点就收不到（Append Entries），这样其它结点过了Election timeout时间，就直接变成候选人，自己进行选举

随后C上位，变成leader。

当然有可能B、C同时变成候选人，可能出现 分裂投票：

如上图，结点B、D同时进入候选人状态，都向临近的三个节点发送投票申请，这时发生分裂投票：

A和C分别投给了D和B，于是B、D票数持平，遇到这种情况系统的解决方案是，原地等待，再等一个Election timeout，直到遇到唯一一个结点进入候选者环节，且唯一胜出。

Log Replication日志同步

大致流程：假设客户端请求想让集群改变状态到“5”，当请求到达leader时，该改变并不会立即提交，先将改变记录存放到leader的log当中，随后由leader结点先分发到各个从结点，随后从结点改变状态，然后修改成功的从结点向leader结点发送成功消息，当leader结点收到大多数成功消息后，就会将这个改动彻底提交，此刻集群的状态变成“5”

Introduction

前面简单介绍了，raft的简洁（相比于Paxos），其具备的特征：

  • Strong Leader：主节点，像日志分发，都是由主节点分发到从结点
  • Leader election：领导选举，在心跳机制上额外添加的机制，用来选择主节点
  • Membership changes：成员变更机制，使用了 joint consensus 算法 approach where the majorities of two different configurations overlap during transitions

本文的其余部分介绍了复制状态机问题(第2节)，讨论了Paxos的优缺点(第3节)，描述了我们的一般可理解性方法(第4节)，介绍了Raft共识算法(第5-8节)，并进行了评估筏(第9节)，并讨论相关工作(第10节)。

Replicated state machines

共识算法通常出现在复制状态机的环境中。在这种方法中，一组服务器上的状态机计算相同状态的相同副本，并且即使某些服务器关闭也可以继续运行。

复制状态机用来解决容错问题：

  • 单个集群结点的大型系统，例如：GFS、HDFS、RAMCLOUD
  • 通常使用一个单独的复制状态机来管理领导者选举 和 存储 必须在领导者崩溃时用来复活的 配置信息。复制状态机的例子包括Chubby和ZooKeeper
  • 算法管理着 记录客户端发来的状态机命令的 日志
  • 状态机处理来自日志的相同命令序列，因此它们产生相同的输出

关键问题：

  1. 日志的一致性如何保证？ 最终每个日志里的命令请求顺序必须一致

分布式系统的共识算法有如下特性：

  • 安全性
  • 高可用：只要大多数服务器能正常运行，整个集群就能正常工作
  • 不依赖时效性来确保日志的一致性：错误的时钟和极端的消息延迟在最坏的情况下会导致可用性问题
  • 响应：大多数集群结点可以完成命令，少数速度较慢的无需影响总体系统性能

  2. 实际构建没有良好的基础

Designing for understandability

设计Raft的目标：

  1. 为系统构建提供一个完整和实用的基础，从而显著减少开发人员所需的设计工作量
  2. 在所有条件下都是安全的，并在典型的操作条件下可用
  3. 普通且一般的操作，它必须是高效的
  4. 具备可理解性
    1. 分解问题
    2. 减少要考虑的状态数量，简化状态空间

The Raft consensus algorithm

Raft将共识问题分解为三个相对独立的子问题：

  • Leader election: 当现有的领导者失败时，必须选择新的领导者(章节5.2)。

  • Log replication : leader必须接受来自客户端的日志条目，并在整个集群中复制它们，迫使其他日志与自己的日志一致(章节5.3)。

    保证属性的一致性：

  • Safety: Raft的关键安全属性是图3中的状态机安全属性:如果任何服务器将特定的日志条目应用到其状态机，那么其他服务器就不能对相同的日志索引应用不同的命令。 第5.4节描述了Raft如何确保此属性;该解决方案涉及对第5.2节中描述的选举机制的额外限制。

Raft Basics

Raft集群包含多个服务器；5个服务器是一个典型的情况，它允许系统容忍其中2个服务器产生故障。在任何给定的时间内，每个服务器都处于三个状态之一：leader、Followers或candidate。

正常的操作下，仅有一个leader和多个followers。

  • Followers是被动的，他们自己不发出请求，只是回应leaders和candidates的请求。
  • leader是主动的，接收客户端请求

candidate是用于选举新的leader

Raft将时间划分为任意长度，如下图所示。

term也就是任期，每一个任期都应该有一个leader当选，每个任期以选举开始，其中一个或多个候选人试图成为leader。如果一位候选人candidate赢得选举，那么他将在接下来的任期时间内担任领导人。Raft保证在给定的任期内最多有一个leader。

不同的服务器，可能会在不同的时间观测到任期之间的过渡，所以某些情况下，某个服务器无法观测到选举阶段乃至整个任期。

任期在Raft中充当逻辑时钟[14]，它们允许服务器检测过时的信息，如过时的leader。每个服务器都有一个任期号，随时间单调增加。服务器进行通信时，都会交换自己的任期：

  • if (服务器A的当前term < 服务器B的) ，则 服务器A的当前term = 服务器B的。
  • if (candidate或leader发现其任期已过)，则立即恢复为follow状态。
  • if (服务器接收到一个带有过期任期号的请求) ，则拒绝该请求。

Raft服务器使用远程过程调用进行通信，而基本共识算法只需要两种类型的rpc：

  • RequestVote rpc由candidate在选举期间发起
  • AppendEntries rpc由leader发起，以复制日志条目并提供心跳形式
  • 第7节添加了第三个RPC，用于在服务器之间传输快照。如果服务器没有及时收到响应，它们会重试rpc，并且它们会并行地发出rpc以获得最佳性能

Leader election

触发选举的机制：心跳机制

  • 服务器只要从leader或candidate接收到有效的rpc请求，就保持follower状态。 因此，leader会定期的向自己的followers发送心跳包（AppendEntries rpc，不携带日志条目），以维护自身的领导地位。

  • 如果服务器在一段时间（elect time out）内，没有收到心跳包，他会假设目前没有leader，于是开始进行新的一轮选举，选择新leader

开始选举

follow增加其当前任期并过渡到candidate状态，然后，它为自己投票，并并行地向集群中的每个其他服务器发出RequestVote rpc。candidate将继续处于这种状态，直到发生以下三种情况之一:

  • 它赢得了选举
  • 另一个服务器确立了自己的领导地位
  • 一段时间过去了，没有人选举胜出

赢得选举

条件：收到大多数服务器的投票。

投票规则：每个服务器在给的的term内，以先到先得的方式投票给最多一个candidate，多数决定原则确保最多有一名candidate能够赢得特定任期的选举。candidate一旦赢得选举，就成为leader。然后，它向所有其他服务器发送心跳消息，以建立其权威并防止新的选举。

其它服务器确立了自己的领导地位

candidate可能会收到一份来自另一个服务器的RPC声称是leader。

  • 如果leader的term(包含在其RPC中) >= candidate's term ，则candidate将承认leader是合法的，并返回到follow状态。
  • 如果leader的term(包含在其RPC中) < candidate's term ，则candidate拒绝RPC并继续处于candidate状态。

没人胜出

许多follow同时成为candidate，选票被分割，这样没有candidate获得多数票。这种情况candidate将超时，并通过增加其任期和启动另一轮RequestVote rpc来开始新的选举。

Split Votes

采用随机选举超时减少分裂投票，candidate的选举超时时间，在某一固定间隔内随机选取（例如：150~300ms）。这将分散服务器（所有的服务器票数相同），因此在大多数情况下，只有一个服务器会超时，它赢得了选举，并在任何其他服务器超时之前发送心跳。

用同样的机制处理分裂投票。每个candidate在选举开始时重新启动其 随机选举超时 ，并等待该超时结束后再开始下一次选举;这降低了在新选举中再次出现分裂投票的可能性。

Log replication

工作流程

  1. 接收请求：一旦选出领导者，它就开始服务客户端请求。每个客户端请求都包含一个由复制状态机执行的命令。

  2. 复制日志：leader把命令作为新条目追加到其日志中，然后并行地向其他每个服务器发出AppendEntries rpc以复制该条目。

  3. 状态机执行命令：条目被安全复制后，leader将条目应用于状态机，将状态机执行的结果封装返回给客户端

  4. follow故障处理：如果follow崩溃或运行缓慢，leader将无限期地重试AppendEntries rpc(即使在它响应客户端之后)，直到所有follow最终存储所有日志条目。

  • 每条日志由一个个条目组成，每个条目包含 使用时的任期 、 用于状态机的命令 、整数索引
  • 任期编号：用于检测日志之间的不一致性，并确保图3中的一些属性
  • 整数索引：标识其条目在日志中的位置

安全的日志条目--commited

leader决定任何时候向状态机应用日志条目都是安全的，这种条目称为 commited。只有当集群中大多数服务器复制了该条目，这个条目才能被提交。与此同时，leader日志中之前的日志也会被提交.

leader会跟踪它知道要提交的最高索引，并且把这个索引包含在接下来的AppendEntries RPCs中（包括心跳），以便其他服务器最终发现。一旦 follow 了解到日志条目已提交，它将该条目应用于其本地状态机(按日志顺序)。

  • 持久性：Raft保证提交的条目是持久的，并且最终将由所有可用的状态机执行

注意：要保证提交的日志任期和leader所在的任期一致。

日志匹配属性：一致性检查

  • 如果不同日志中的两个条目具有相同的索引和任期，则它们存储相同的命令。
  • 如果不同日志中的两个条目具有相同的索引和任期，则它们先前所有条目的日志都相同。

第一个属性源于这样一个事实，即leader在给定的任期内最多创建一个具有给定日志索引的条目，并且日志条目永远不会改变其在日志中的位置。

第二个属性由AppendEntries执行的 简单一致性检查保证。当leader发送AppendEntries RPC时，leader会将前一个日志条目的索引位置和任期号包含在里面，如果follower在其日志中没有找到具有相同索引和任期的条目，则拒绝新条目。

  • 一致性检查作为一个归纳步骤：日志的初始状态为空，肯定满足 “日志匹配属性” ，一致性检查保证了日志扩展过程中的日志匹配特性，因此，每当AppendEntries成功返回时，leader就知道follower的日志与它自己的日志在添加新条目之后完全相同。

leader崩溃所导致的日志不一致

leader 崩溃的情况会使日志处于不一致的状态，old leader 可能还没有完全复制它日志里的所有条目，这种不一致会在一系列的 leader 和 follower 崩溃的情况下加剧。

图 7 展示了在什么情况下 follower 的日志可能和新的 leader 的日志不同：

  • Follower 可能缺少一些在新 leader 中有的日志条目，
  • 也可能拥有一些新 leader 没有的日志条目，或者与上一条同时发生。
  • 缺失或多出日志条目的情况可能会涉及到多个任期。

  1. 当一个 leader 成功当选时（最上面那条日志），follower 可能是（a-f）中的任何情况。

  2. 每一个方格表示一个日志条目，里面的数字表示任期号。如下情况，follower日志与leader不一致

    • Follower 可能会缺少一些日志条目（a-b），

    • 可能会有一些未被提交的日志条目（c-d），

    • 或者两种情况都存在（e-f）。

例如，场景 f 可能会这样发生：

  1. f 对应的服务器在任期 2 的时候是 leader ，追加了一些日志条目到自己的日志中，一条都还没提交（commit）就崩溃了
  2. 该服务器很快重启，在任期 3 重新被选为 leader，又追加了一些日志条目到自己的日志中；
  3. 在任期 2 和任期 3 中的日志都还没被提交之前，该服务器又宕机了，并且在接下来的几个任期里一直处于宕机状态。

解决不一致

leader 通过强制 follower 复制它的日志来解决不一致的问题。这意味着 follower 中跟 leader 冲突的日志条目会被 leader 的日志条目覆盖。

如何复制？

  1. leader 必须找到两者达成一致的索引最大的日志条目（假设为x）
  2. 删除follower 日志中x后面的全部条目
  3. 将自己x后面的日志条目发给follower

以上操作都封装在对 AppendEntries RPCs 中一致性检查的回复中：Leader为每个followers维护一个nextId，表示要发送给followers的下一个条目的索引。当选出一个新 leader 时，该 leader 将所有 nextIndex 的值都初始化为自己最后一个日志条目的 index 加1（图7中的11）。如果follower 的日志和 leader 的不一致，那么 AppendEntries RPC 中的一致性检查就会失败，follower就会拒绝接收，随后leader会减少nextIndex，并重试AppendEntries RPCs ，重试一定次数以后，最后nextIndex会在leader与follower的日志达成一致的地方，此时AppendEntries RPCs 就会成功，follower中跟 leader 冲突的日志条目全部删除然后追加 leader 中的日志条目，在接下来的时间里都保持一致。

优化

优化方向：减少AppendEntries RPC的次数。

当拒绝一个 AppendEntries RPC 的请求的时候，follower 可以包含冲突条目的任期号和自己存储的那个任期的第一个 index 。借助这些信息，leader 可以跳过那个任期内所有冲突的日志条目来减小 nextIndex；

机制的好处

通过这种机制，leader 在当权之后就不需要任何特殊的操作来使日志恢复到一致状态。Leader 只需要进行正常的操作，然后日志就能在回复 AppendEntries rpc 一致性检查失败的时候自动趋于一致。

Leader 从来不会覆盖或者删除自己的日志条目（图 3 的 Leader Append-Only 属性）。

这样的日志复制机制展示了其一致性特性：只要过半的服务器能正常运行，Raft 就能够接受，复制并应用新的日志条目；在正常情况下，新的日志条目可以在一个 RPC 来回中被复制给集群中的过半机器；并且单个运行慢的 follower 不会影响整体的性能。

Safety

前面的部分描述了 Raft 如何选举领导者和复制日志条目。然而，到目前为止所描述的机制还不足以确保每个状态机以相同的顺序执行完全相同的命令。例如，当领导者提交多个日志条目时，follower 可能不可用，然后它可以被选为领导者并用新的条目覆盖这些条目；因此，不同的状态机可能会执行不同的命令序列。 这一章 完成 Raft 算法：增加限制到： 哪些 server 可能被选为 leader。限制确保：任意给定的 term 的 leader 包含所有之前 term 已经提交的元素（图 3），根据选举限制，我们使得 规则更加准确。我们给出一个 leader 完整性的 图 并展示他如何纠正状态机的行为。

Election restriction

原则：在任何 leader-based 一致性算法中,leader必须最终存储 所有committed 日志元素。

Raft保证从选举的那一刻起，每个新领导者都存在 之前 任期的所有已提交条目，无需自己在更新之前的条目，这保证了日志条目流动的单向性：从leader -> followers 。

限制规则：Raft 在 投票流程 去保证候选人只有在它的 log 中包含所有已经提交的日志，才能从选举中胜出

如何判断候选人在它的 log 中包含所有已经提交的日志：候选人请求投票的时候必须联系集群中的大部分节点，候选人的日志至少与该多数人中的任何其他日志一样最新，才被判定符合

实现：

RequestVote RPC 实现了这个限制：

  • RPC 包含关于候选人日志的信息
  • 投票者自己的日志比候选人的日志 更新，则投票者拒绝投票。

通过比较两者日志中最后一个条目的索引和任期来确定两个日志中哪个是最新的：

  • 不同term：具有较大的term号的日志是最新的
  • 相同term：日志较长的就是最新的

Committing entries from previous terms

问题发现

一旦当前任期内的某个日志条目已经存储到过半的服务器节点上，leader 就知道该日志条目已经被提交了。如果 leader 在提交一个 entry 之前 crash 了，新的 leader 将尝试 完成这个 entry 的复制；

然而， 如果是之前任期内的某个日志条目已经存储到过半的服务器节点上，leader 也无法立即断定该日志条目已经被提交了。图 8 展示了一种情况，一个已经被存储到过半节点上的老日志条目，仍然有可能会被未来的 leader 覆盖掉。

  1. （a）S1 是 leader 并且部分复制了 log 到 index 2
  2. （b）S1 crash；S5 被选为term3 的 leader（s3，s4，和 s5 的投票）；并且接受了不同于 index2 的位置的 entry ；
  3. （c）s5 crash，s1 重启，被选为 leader，并且继续复制。此时， term 2的 log entry 已经被复制到了大多数 server 上，但是还没有提交
  4. （d）S1 crash ，S5 可以被选为 leader（S2，S3，S4 投票）并且使用 term3 的数据覆盖了 entry
  5. （e）相比于（d），如果 S1 crash 前复制一个 entry （在大多数 server 上）从当前 term。然后 entry 被提交（S5 就不可能选举成功）。在这种情况下，之前的所有日志('2')也被提交了。

d或者e都有可能产生，所以“leader 也无法立即断定该日志条目已经被提交了”

解决

为了排除 图 8 的问题，Raft 从不通过计算副本数量来提交之前任期的日志条目，只有 leader 当前 term 内的日志条目才通过计算副本数目的方式来提交；一旦当前 term 的某个日志条目以这种方式被提交，那么由于日志匹配特性，之前的所有日志条目也都会被间接地提交。在某些情况下，领导人可以安全地断定一个老的日志条目已经被提交（例如，如果该条目已经存储到所有服务器上），但是 Raft 为了简化问题使用了一种更加保守的方法。

Raft 会在提交规则上增加额外的复杂性：因为当 leader 复制之前任期内的日志条目时，这些日志条目都保留原来的任期号。在其他的一致性算法中，如果一个新的 leader 要重新复制之前的任期里的日志时，它必须使用当前新的任期号。Raft 的做法使得更加容易推导出（reason about）日志条目，因为他们自始至终都使用同一个任期号。另外，和其他的算法相比，Raft 中的新 leader 只需要发送更少的日志条目（其他算法中必须在它们被提交之前发送更多的冗余日志条目来给它们重新编号）。

Safety argument

有了完整的Raft算法，Leader Completeness属性可以被更精确的论证(这个论证基于9.2节的安全保证)。

假设Leader Completeness Property不成立，然后推导出矛盾：假如任期T的leader（T）提交了一条日志x，但是这个日志没有被未来任期的leader储存。例如任期U>T的leader(U)没有储存这个日志条目x。

  1. U在选举的时候日志中没有该条目(leader不会删除或者覆盖它自己的日志条目)

  2. T把这条日志复制到集群大部分服务器上，U收到了集群大多数服务器的投票。因此，至少一个服务器不仅接收了T的日志条目，还给U投票了，如图9所示的S3。这个服务器就是矛盾出现的关键。

  • 如果任期T的leader S1在它的任期提交了一条日志，S5是新任期U选举的leader，那么肯定有一个S3既接受了日志又给S5投票。
  1. 服务器在给U投票之前必须接受T通过AppendEntries rpc提交的日志条目；否则它会拒绝T的AppendEntries rpc请求，因为它当前的任期可能比T要高
  2. 服务器投票给U时，仍然存储该条目，因为每个leader都包含该条目，leader从不删除条目，而follower仅在与leader发生冲突时才删除条目。
  3. 服务器给U投票，所以U的日志中必须已经和服务器的一样新(up-to-date)。这会导致下面两个矛盾中的一种。
  4. 首先，如果服务器的上一任期号和U相同，那么U的日志必须至少与服务器的日志一样新，因此U的日志包含选民日志中的每个条目。这是自相矛盾的，因为选民中有提交的条目，而U在假设中没有该条目x。
  5. 另外，U的最高日志条目的任期必须大于服务器的任期。 而且它大于T，因为服务器的最高日志的任期至少为T（包含T任期中提交的日志条目）。 创建U的最高日志条目的前任leader必须已在其日志中包含提交的条目（假设）。 然后，通过Log Matching属性，U的日志还必须包含已提交的条目，这是矛盾的。
  6. 这样就有了以上所有矛盾。 因此，所有任期T之后的leader必须包含任期T中提交的所有日志条目。
  7. Log Matching属性保证未来的leaders间接包含提交的日志条目，如图8的索引为2的条目。

通过Leader Completeness属性，可以证明图3中的State Machine Safety属性，这个特性的内容是：如果服务器已将给定索引的日志条目应用于其状态机，则其他任何服务器在该索引处都不会应用不同的日志条目。服务器将日志条目应用于其状态机时，其日志必须与通过该条目的leader的日志相同，并且必须提交该条目。现在考虑任何服务器应用给定日志索引的最低任期。Log Completeness属性可确保所有更高任期的leader都将存储相同的日志条目，之后任期中应用该索引处条目的服务器将应用相同的值。因此，State Machine Safety属性成立。最后，Raft要求服务器以日志索引的顺序应用条目。结合State Machine Safety属性，这意味着所有服务器将以相同的顺序将完全相同的日志条目集合应用于其状态机。

Follower and candidate crashes

follower和candidate崩溃比leader崩溃更容易处理，并且它们都以相同的方式处理。

follower或者candidate崩溃，将来发送给它的RequestVote和AppendEntries RPC将失败。Raft通过无限期重试来处理这些故障；

  • 如果崩溃的服务器重新启动，则RPC将成功完成。
  • 如果服务器在完成RPC之后但在响应之前崩溃了，则重新启动后它将再次收到相同的RPC。

Raft的RPC是幂等的，因此不会造成问题。 例如，follower收到一个AppendEntries请求，内容是已经保存的日志条目，它会忽略这些条目。

Timing and availability

对Raft的要求之一是安全性不得依赖于运行的时间顺序：系统不得由于某些事件比预期的发生得更快或更慢而产生错误的结果。 但是，可用性（系统及时响应客户端的能力）必然取决于时间。

  • 例如，如果信息传递花费的时间比服务器两次宕机之间稳定运行的时间要长，candidate没有足够的时间选举；就会没有稳定的leader，Raft无法执行任何操作。

leader选举是Raft最重要的方面，时间至关重要。只要系统满足timing requirement，Raft将能够选举和维持稳定的leader：

  • broadcastTime是服务器与集群中的每个服务器并行发送RPC并接收其响应所花费的平均时间；
  • electionTimeout是5.2节中描述的选举超时；
  • MTBF是单个服务器两次故障之间的平均时间。

broadcastTime应比electionTimeout小一个数量级，以便leader可以可靠地发送心跳消息，而且防止follower开始选举；考虑到使用随机方法实现选举超时，也使分裂投票变得不太可能。electionTimeout应比MTBF小几个数量级，以使系统稳定运行。当leader崩溃时，该系统将在大约electionTimeout时间内不可用；我们希望这只代表总时间的一小部分。

broadcastTime和MTBF是底层系统的属性，而elelctionTimeout是我们必须选择的东西。 Raft的RPC通常需要接收者将信息持久化，因此broadcastTime可能从0.5毫秒到20毫秒不等，具体取决于存储技术。因此，electionTimeout可能在10毫秒至500毫秒之间。典型服务器的MTBF一般是几个月甚至更长，很容易就满足了timing requirement。

Log compaction

随着日志量的无限增加，最终会刷满磁盘。影响可用性。通过快照存储形式，删除一部分日志是比较好的解决方案。

如上图，将前五条日志，包装成快照，只记录状态机在这五个日志中最后最新的状态，然后leader维护lastIncludedIndex和lastIncludedTerm表示上一次快照后，快照涉及的最新的那条日志。

具体的压缩步骤：

  1. 将节点的状态保存为LSM Tree,然后存储最后应用日志的索引与任期,以保证日志匹配特性
  2. 为支持集群的配置更新,快照中也要将最后应用的集群配置也当做状态保存下来
  3. 当跟随者需要的日志已经在领导者上面被删除时(nextIndex--),需要将快照通过RPC发送过

实现细节：

  1. 如果term < currentTerm就立即拒绝
  2. 如果是第一个分块 (offset 为 0 ) 就创建一个新的快照
  3. 在指定偏移量写入数据
  4. 如果 done 是 false，则继续等待更多的数据 ack
  5. 保存快照文件，丢弃具有较小索引的任何现有或部分快照
  6. 如果现存的日志条目与快照中最后包含的日志条目具有相同的索引值和任期号，则保留其后的日志条目并进行回复
  7. 否则 丢弃整个日志
  8. 使用快照重置状态机(并加载快照的集群配置)

some questions

快照何时创建？

如果用定时器的方式，需要设置一个恰当的阈值，过大过小都不行。

我们可以采用redis中的方式，就是定一个空间大小的阈值，超过这个阈值了就直接创建。

快照花费时间怎么办？

异步创建，状态机的函数式顺序性天然支持。

优缺点：

  • 快照设置频繁下带来的浪费磁盘带宽和能源问题。当然快照设置不过与频繁也会导致存储问题。因此合理的安装快照才是正确的解决办法。 Paper中提到的策略就是对Log设置一个合理的大小，超过这个大小就进行日志快照的更新

  • 另外的就是在写入快照时进行的时间开销。因为可能在写入快照时会有竞争，并加锁。paper中提到的也是经典copy-on-write方法，在更新快照时，先临时储存一份log的备份
