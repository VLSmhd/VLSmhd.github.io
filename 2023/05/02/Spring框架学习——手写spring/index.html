创建简单的bean容器：

概念：

spring包含并管理应用对象的生命周期和配置，spring容器是承载应用对象的载体。

我们可以自行定义bean，每用到应用对象，都会进行“bean的加载，对象创建、对象实例化等等”。bean中也会用到其它的bean。

我们将所有的bean交给spring统一管理，在业务进行的时候由spring帮我们创建对象，管理对象的生命周期。

容器：能存放数据的具体数据结构都叫容器。如：ararayList hashMap等等。

对于spring而言，每个bean都有其独一无二的名称，在应用中会频繁的获取、销毁bean，我们采用哈希表hashmap再合适不过了。

目标：

我们在上一部分写的bean的注册逻辑非常简单，而且仅仅只是注册了类信息，而不会把类的实例化信息注册到spring。

把 Object bean 改为 Class bean

  • BeanFactory工厂的完善：把beanFactory定义为接口，用AbstractBeanFactory抽象类实现，使用模板模式的设计方式，可以统一归纳通用核心方法的调用逻辑和标准定义，也就很好的控制了后续的实现者不用关心调用逻辑，按照统一方式执行。那么类的继承者只需要关心具体方法的逻辑实现即可。

  • 在继承抽象类 AbstractBeanFactory 后的 AbstractAutowireCapableBeanFactory 就可以实现相应的抽象方法了，因为 AbstractAutowireCapableBeanFactory 本身也是一个抽象类，所以它只会实现属于自己的抽象方法，其他抽象方法由继承 AbstractAutowireCapableBeanFactory 的类实现。这里就体现了类实现过程中的各司其职，你只需要关心属于你的内容，不是你的内容，不要参与。

  • 关于单例 SingletonBeanRegistry 的接口定义实现，而 DefaultSingletonBeanRegistry 对接口实现后，会被抽象类 AbstractBeanFactory 继承。现在 AbstractBeanFactory 就是一个非常完整且强大的抽象类了，也能非常好的体现出它对模板模式的抽象定义。

基于Cglib实现含构造函数的类实例化策略

问题描述：

上一节，我们获取bean对象的方法是createBean，通过反射实例化bean对象。

但是如果bean里有含参构造函数，就会抛出异常：

原因：

使用newInstance方法实例化含有有参构造函数的类对象会报错，是因为newInstance方法会调用类的无参构造函数来创建实例。如果一个类只有有参构造函数而没有无参构造函数，那么newInstance方法无法创建实例，因为它没有提供构造函数参数。

目标：

  1. 从哪合理的把构造函数的入参信息传递到实例化操作？
  2. 怎样实例化含有有参构造函数的对象？

  • 参考 Spring Bean 容器源码的实现方式，在 BeanFactory 中添加 Object getBean(String name, Object... args) 接口，这样就可以在获取 Bean 时把构造函数的入参信息传递进去了。

  • 基于 Java 本身自带的方法 DeclaredConstructor，另外一个是使用 Cglib 来动态创建 Bean 对象。Cglib 是基于字节码框架 ASM 实现，所以你也可以直接通过 ASM 操作指令码来创建对象

    • DeclaredConstructor：是Java反射API中的一个类，表示类的构造函数对象。它继承自Constructor类，因此包含了Constructor类的所有方法，同时也提供了一些额外的方法用于获取构造函数的信息。

      在Java中，每个类都有一个或多个构造函数，用于创建该类的对象。DeclaredConstructor类封装了这些构造函数，允许在运行时通过反射来获取构造函数的详细信息，如参数类型、访问修饰符等。

      常用方法：

      • newInstance(Object... initargs)：使用构造函数来创建一个新的对象，其中initargs是一个可变参数列表，表示构造函数的参数。
      • setAccessible(boolean flag)：设置构造函数的可访问性，如果设置为true，则可以访问私有构造函数，否则不能。
      • getModifiers()：获取构造函数的访问修饰符，返回一个表示修饰符的整数。
      • getParameterTypes()：获取构造函数的参数类型，返回一个表示参数类型的Class对象数组。

    • Cglib：是一个基于ASM开发的高性能字节码增强库，可以在运行时动态地生成指定类的子类，并覆盖其中的方法。因此，Cglib可以实现类似于Java动态代理的功能，同时还可以对类进行更灵活的增强操作。

      Cglib中实例化一个bean对象的过程通常分为以下几步：

      1. 创建Enhancer对象，用于生成子类。
      2. 设置父类和回调方法。
      3. 使用create()方法创建子类对象。

设计：

功能设计结构：
  • 资源加载器属于相对独立的部分，它位于 Spring 框架核心包下的IO实现内容，主要用于处理Class、本地和云环境中的文件信息。

前置知识：
spring的classpath（类路径）作用：

Spring通过ClassPath来加载和管理应用程序的类和资源。

在Spring中，ClassPath的概念主要用于以下几个方面：

  1. 类加载：Spring使用ClassLoader来加载应用程序的类。在Spring的上下文中，可以配置类加载器的ClassPath来指定加载类文件的路径。这样，Spring就能够根据ClassPath来加载应用程序中的各种类。
  2. 资源加载：Spring提供了资源加载的功能，可以通过ClassPath来加载类路径下的资源文件。这些资源文件可以是配置文件、XML文件、属性文件等。通过ClassPath资源加载，Spring能够方便地获取并管理应用程序所需的各种配置文件和资源。
  3. 包扫描：Spring框架支持自动扫描应用程序中的类，并根据类路径来发现和注册Bean。通过配置ClassPath的范围，可以指定Spring在哪些路径下进行包扫描，以及需要扫描哪些包和类。

例如：

classpath:config.xml ：表示在类路径下查找名为config.xml的文件。

实现应用上下文，自动识别、资源加载、扩展机制

目标：

  • 加载策略进行优化

    我们不太可能让面向 Spring 本身开发的 DefaultListableBeanFactory 服务，直接给予用户使用。修改点如下：

  • 把 Bean 对象扩展机制功能和对 Spring 框架上下文的包装融合起来，对外提供完整的服务。

设计：

  • 满足于对 Bean 对象扩展的两个接口，其实也是 Spring 框架中非常具有重量级的两个接口：、

    BeanFactoryPostProcessor 和 BeanPostProcessor，也几乎是大家在使用 Spring 框架额外新增开发自己组建需求的两个必备接口。

    • BeanFactoryPostProcessor：由 Spring 框架组建提供的容器扩展机制，允许在 Bean 对象注册后但未实例化之前，对 Bean 的定义信息 BeanDefinition 执行修改操作。

    • BeanPostProcessor，也是 Spring 提供的扩展机制，不过 BeanPostProcessor 是在 Bean 对象实例化之后修改 Bean 对象，也可以替换 Bean 对象。

ApplicationContext：

  • context 是本次实现应用上下文功能新增的服务包
  • ApplicationContext，继承于 ListableBeanFactory，也就继承了关于 BeanFactory 方法，比如一些 getBean 的方法。另外 ApplicationContext 本身是 Central 接口，但目前还不需要添加一些获取ID和父类上下文，所以暂时没有接口方法的定义。

ConfigurableApplicationContext：

是Spring框架中的一个接口，用于表示一个可配置的应用程序上下文。它是ApplicationContext接口的子接口，提供了额外的配置和管理应用程序上下文的功能。

AbstractApplicationContext：

  • AbstractApplicationContext 继承 DefaultResourceLoader 是为了处理 spring.xml 配置资源的加载。
  • 之后是在 refresh() 定义实现过程，包括：
    • 创建 BeanFactory，并加载 BeanDefinition
    • 获取 BeanFactory
    • 在 Bean 实例化之前，执行 BeanFactoryPostProcessor (Invoke factory processors registered as beans in the context.)
    • BeanPostProcessor 需要提前于其他 Bean 对象实例化之前执行注册操作
    • 提前实例化单例Bean对象
  • 另外把定义出来的抽象方法，refreshBeanFactory()、getBeanFactory() 由后面的继承此抽象类的其他抽象类实现。

AbstractRefreshableApplicationContext：获取Bean工厂和加载资源

也是一个模板方法，实现了AbstractApplicationContext的refreshBeanFactory抽象方法，留了loadBeanDefinitions给后代实现。

  • 在 refreshBeanFactory() 中主要是获取了 DefaultListableBeanFactory 的实例化以及对资源配置的加载操作 loadBeanDefinitions(beanFactory)，在加载完成后即可完成对 spring.xml 配置文件中 Bean 对象的定义和注册，同时也包括实现了接口 BeanFactoryPostProcessor、BeanPostProcessor 的配置 Bean 信息。
  • 但此时资源加载还只是定义了一个抽象类方法 loadBeanDefinitions(DefaultListableBeanFactory beanFactory)，继续由其他抽象类继承实现。因为加载方式的不同，可能需要不同类来扩展。

AbstractXmlApplicationContext：上下文中对配置信息的加载

继承AbstractRefreshableApplicationContext类，实现loadBeanDefinitions方法，提供xml加载方式。同时自身也作为模板，给后代方法getConfigLocations，来获取资源配置的位置信息。
